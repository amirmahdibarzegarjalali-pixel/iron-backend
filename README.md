# بک‌اند پروتکل آیرون — فاز ۱ (حساب کاربری، احراز هویت، پروفایل)

این فاز اول از تبدیل «پروتکل آیرون» به یک محصول واقعی است: یک بک‌اند واقعی، تست‌شده
و آماده‌ی اجرا که حساب کاربری، احراز هویت امن و پروفایل کاربران را روی سرور
مدیریت می‌کند (به‌جای localStorage در مرورگر). فازهای بعدی (VIP/پرداخت، پنل
مدیریت، هوش مصنوعی، گیمیفیکیشن، ...) روی همین پایه ساخته می‌شوند.

## چرا بدون Express/Prisma نوشته شده؟

عمداً **بدون هیچ وابستگی خارجی** نوشته شده (فقط ماژول‌های داخلی Node.js:
`http`، `crypto`، `fs`). دلیلش:

- می‌توانید همین الان، بدون `npm install` و بدون سرور، آن را اجرا و تست کنید.
- هیچ ریسک امنیتی زنجیره‌ی تأمین (supply chain) از پکیج‌های شخص ثالث ندارد.
- روی هر سرویس میزبانی Node (حتی رایگان‌ترین‌ها) بدون دردسر بالا می‌آید.
- اگر بعداً خواستید به Express/Fastify مهاجرت کنید، چون معماری (روتر/میان‌افزار/
  کنترلر) از همان الگو پیروی می‌کند، مهاجرت ساده خواهد بود.

دیتابیس هم به همین دلیل، به‌صورت پیش‌فرض یک datastore سبک فایل‌محور (JSON) است
که **کاملاً واقعی و پایدار** است (نه شبیه‌سازی)، اما برای تولید در مقیاس
بزرگ باید به PostgreSQL مهاجرت کنید — که مسیرش کاملاً آماده شده (پایین را ببینید).

## اجرای سریع (بدون سرور، روی سیستم خودتان)

```bash
cd iron-backend
cp .env.example .env      # و مقادیر JWT_SECRET را عوض کنید
node src/server.js
# سرور روی http://localhost:4000 بالا می‌آید
```

تست کامل (۳۹ سناریوی end-to-end، شامل ثبت‌نام، ورود، رفرش توکن، آپلود عکس،
OTP پیامکی، بازیابی رمز، ریت‌لیمیت، حذف حساب):

```bash
node src/server.js &     # سرور را در پس‌زمینه اجرا کنید
node tests/auth.e2e.test.js
```

## ساختار پروژه

```
iron-backend/
├── src/
│   ├── server.js              # نقطه‌ی ورود
│   ├── router.js              # روتر سبک (Express-like) بدون وابستگی
│   ├── config/env.js          # خواندن متغیرهای محیطی
│   ├── db/
│   │   ├── jsonStore.js       # دیتابیس فایل‌محور فعلی
│   │   └── repositories/      # لایه‌ی دسترسی به داده (قابل تعویض)
│   │       ├── userRepository.js
│   │       ├── sessionRepository.js
│   │       ├── tokenRepository.js
│   │       ├── securityLogRepository.js
│   │       └── postgres/
│   │           ├── schema.sql          # طرح کامل دیتابیس (همه‌ی فازها)
│   │           └── userRepository.pg.js # نمونه‌ی معادل روی PostgreSQL
│   ├── modules/
│   │   ├── auth/     # ثبت‌نام، ورود، OTP، رفرش، بازیابی رمز
│   │   └── users/    # پروفایل، آپلود عکس، اندازه‌گیری بدن، رکوردها
│   ├── middleware/    # auth (JWT)، rateLimit
│   └── utils/         # jwt.js، hash.js، mailer.js، sms.js، multipart.js، ...
├── tests/auth.e2e.test.js
├── uploads/            # عکس پروفایل کاربران
├── data/               # فایل‌های دیتابیس JSON (در گیت نادیده گرفته شود)
└── .env.example
```

معماری **ماژولار و لایه‌بندی‌شده** است: `routes → controller → service → repository`.
هیچ منطق تجاری در روت‌ها یا کنترلرها نیست؛ همه در `*.service.js` است تا تست و
توسعه‌ی آینده ساده بماند.

## API پیاده‌سازی‌شده

| متد | مسیر | توضیح | نیاز به توکن |
|---|---|---|---|
| POST | `/api/auth/register` | ثبت‌نام با ایمیل/شماره + رمز | ❌ |
| POST | `/api/auth/login` | ورود با ایمیل یا شماره + رمز | ❌ |
| POST | `/api/auth/otp/request` | درخواست کد ورود پیامکی | ❌ |
| POST | `/api/auth/otp/verify` | تایید کد و ورود | ❌ |
| POST | `/api/auth/refresh` | تمدید accessToken با refreshToken | ❌ |
| POST | `/api/auth/logout` | خروج از یک دستگاه | ❌ |
| POST | `/api/auth/logout-all` | خروج از همه‌ی دستگاه‌ها | ✅ |
| POST | `/api/auth/forgot-password` | ارسال لینک بازیابی رمز | ❌ |
| POST | `/api/auth/reset-password` | ثبت رمز جدید | ❌ |
| GET | `/api/users/me` | دریافت پروفایل | ✅ |
| PUT | `/api/users/me` | ویرایش پروفایل | ✅ |
| POST | `/api/users/me/avatar` | آپلود عکس پروفایل (multipart) | ✅ |
| DELETE | `/api/users/me` | حذف حساب (نیاز به تایید رمز) | ✅ |
| POST/GET | `/api/users/me/body-stats` | ثبت/دریافت اندازه‌گیری بدن (نمودار پیشرفت) | ✅ |
| POST/GET | `/api/users/me/prs` | ثبت/دریافت رکوردهای شخصی | ✅ |

توکن با هدر `Authorization: Bearer <accessToken>` ارسال می‌شود.

### امنیت پیاده‌سازی‌شده
- هش رمز با `scrypt` (داخلی Node، امنیت معادل bcrypt)
- JWT با HS256، انقضای کوتاه برای accessToken (۱۵ دقیقه پیش‌فرض) + refreshToken
  چرخشی (rotate) که در پایگاه‌داده قابل ابطال است (خروج واقعی، نه فقط سمت کلاینت)
- Rate limiting روی مسیرهای حساس (ورود، ثبت‌نام، فراموشی رمز، OTP)
- اعتبارسنجی کامل ورودی‌ها سمت سرور
- لاگ رویدادهای امنیتی (`security_events`): ورود موفق/ناموفق، تغییر رمز، حذف حساب
- عدم افشای وجود/عدم‌وجود ایمیل در «فراموشی رمز» (پاسخ همیشه یکسان است)
- هدرهای امنیتی پایه (X-Content-Type-Options، X-Frame-Options، ...)

## چون سرور ندارید — راهنمای میزبانی

برای این فاز (بدون نیاز به دیتابیس جداگانه چون از JSON استفاده می‌کند)، ساده‌ترین
گزینه‌ها یک PaaS رایگان/ارزان با پشتیبانی از Node.js است:

1. **Render.com** یا **Railway.app**: ریپازیتوری گیت را وصل کنید، دستور اجرا را
   `node src/server.js` بگذارید، متغیرهای `.env` را در پنل تنظیم کنید. هر دو یک
   دامنه‌ی رایگان `https://...onrender.com` می‌دهند.
2. یک **VPS ارزان** (مثلاً آبان یا آروان): Node را نصب کنید، پروژه را بالا بیاورید
   و با `pm2` یا `systemd` همیشه روشن نگه دارید، پشت Nginx با SSL رایگان (Let's Encrypt).

⚠️ نکته‌ی مهم درباره‌ی دیتابیس JSON: روی اکثر PaaSها فایل‌سیستم بین دیپلوی‌ها
پاک می‌شود (ephemeral). یعنی برای تولید واقعی (نه فقط تست)، باید یا:
- یک **Persistent Disk/Volume** به سرویس میزبانی وصل کنید (Render و Railway هر دو
  این قابلیت را دارند)، یا
- به PostgreSQL مهاجرت کنید (پیشنهاد اصلی برای فاز بعد — پایین را ببینید).

## مهاجرت به PostgreSQL (وقتی آماده‌ی مقیاس واقعی بودید)

1. یک دیتابیس PostgreSQL بگیرید (مثلاً Neon.tech یا Supabase — هر دو رایگان شروع می‌شوند).
2. `schema.sql` را روی آن اجرا کنید (شامل جدول‌های همه‌ی فازهای آینده هم هست).
3. `npm install pg`
4. `DATABASE_URL` را در `.env` بگذارید.
5. در فایل‌های `src/db/repositories/*.js`، به‌جای `require("../jsonStore")` از
   معادل PostgreSQL (نمونه در `postgres/userRepository.pg.js`) استفاده کنید.

⚠️ **نکته‌ی صادقانه:** نسخه‌ی JSON فعلاً synchronous است (بدون async/await) و
نسخه‌ی PostgreSQL به‌طور طبیعی async است (چون کوئری روی شبکه می‌رود). یعنی هنگام
سوییچ باید در فایل‌های سرویس (`auth.service.js`, `users.service.js`) جلوی
فراخوانی‌های ریپازیتوری، `await` اضافه کنید. امضای متدها و نام‌ها یکسان می‌ماند،
اما این یک تغییر کد کوچک (نه بازنویسی کامل) لازم دارد — این را شفاف می‌گویم چون
قول «هیچ تغییری لازم نیست» گمراه‌کننده بود.

## نقشه‌ی راه فازهای بعدی (طبق درخواست کامل شما)

طرح دیتابیس همه‌ی این‌ها از الان در `schema.sql` آماده است؛ فقط پیاده‌سازی کد باقی مانده:

- **فاز ۲ — VIP و پرداخت:** پلن‌ها، لایه‌ی abstraction پرداخت (اول زرین‌پال، بعد
  مایکت/بازار با همان اینترفیس)، فعال‌سازی خودکار VIP، کد تخفیف.
- **فاز ۳ — پنل مدیریت:** یک اپ فرانت‌اند جدا (React) روی همین APIها + مسیرهای
  admin (`/api/admin/...`) با نقش‌بندی (role-based access).
- **فاز ۴ — هوش مصنوعی:** endpoint مربی هوشمند با اتصال به Claude/OpenAI API
  (کلید از پنل مدیریت قابل تغییر)، محدودیت ۵ پیام رایگان روزانه (جدول `ai_usage_daily`
  از الان آماده است)، تحلیل عکس غذا/بدن.
- **فاز ۵ — گیمیفیکیشن:** مدال، XP، سطح، چالش، Referral (جدول‌ها آماده‌اند).
- **فاز ۶ — محتوا و پشتیبانی:** اخبار، مقالات، ویدیو، تیکت.
- **فاز ۷ — تبلیغات و اعلان‌ها:** AdMob (فقط کاربران رایگان)، Push Notification.
- **بسته‌بندی برای استورها:** چون فرانت‌اند فعلی HTML/PWA است، برای گوگل‌پلی/
  بازار/مایکت باید با **Capacitor** به یک اپ Android بسته‌بندی شود (مرحله‌ی جدا،
  می‌توانم در فاز مربوطه انجامش بدهم).

هر فاز را که خواستید شروع کنیم بگویید تا با همین کیفیت (کد واقعی + تست
end-to-end واقعی، نه نمایشی) پیش برویم.
